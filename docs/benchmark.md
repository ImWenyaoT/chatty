# Chatty 性能测量报告

本文档记录 2026-07-23 的一次本机基准和真实 DeepSeek 小样本测试。数字只适用于下述口径。

## 1. 为什么不能直接写 CTR 和可用性

点击率（CTR）需要真实曝光和点击事件：

```text
CTR = 点击次数 / 曝光次数
```

Chatty 没有真实用户流量，因此不能计算推荐 CTR、CTR 提升或转化率。

生产可用性需要持续监控窗口：

```text
Availability = 成功请求数 / 总请求数
```

5 次或 500 次测试只能称为“样本成功率”，不能代表月度生产可用性。

| 项目 | 环境 |
|---|---|
| CPU | Intel Core i7-14700KF，20 核 28 线程 |

## 3. 本机确定性全链路基准

### 测量范围

每个请求完整经过：

```text
FastAPI TestClient
  → Pydantic 请求校验
  → OpenAI Agents SDK Runner
  → 5 次 Tool Calling
  → SQLite 业务查询
  → SQLite FTS5 RAG
  → Pydantic 输出校验
  → HTTP 响应序列化
```

模型使用确定性脚本模型，不包含 DeepSeek 网络和推理时间。测试先预热 20 次，再串行执行 500 次请求。

### 结果

| 指标 | 测量值 |
|---|---:|
| 请求数 | 500 |
| 失败数 | 0 |
| 样本成功率 | 100% |
| 总耗时 | 2.7962 s |
| 吞吐 | 178.82 QPS |
| 平均延迟 | 5.592 ms |
| P50 | 5.496 ms |
| P95 | 5.913 ms |
| P99 | 6.236 ms |
| 最大延迟 | 42.527 ms |

该结果描述本机代码路径，不是生产容量，也不代表接入 DeepSeek 后的吞吐。

## 4. 真实 DeepSeek 端到端采样

### 测量范围

5 次请求按顺序调用 `deepseek-v4-pro`。每次请求都执行完整 Agent Loop、五个 Tool、FTS5 RAG 和最终校验，并返回 3 个商品。

### 结果

| 指标 | 测量值 |
|---|---:|
| 样本数 | 5 |
| 成功数 | 5 |
| 失败数 | 0 |
| 样本成功率 | 100% |
| 最小延迟 | 12.917 s |
| 中位延迟 | 17.247 s |
| 平均延迟 | 17.188 s |
| 最大延迟 | 21.063 s |
| 顺序请求吞吐 | 0.0582 requests/s |

5 个样本不足以计算稳定的 P95 或 P99，也不能代表生产可用性。

## 5. 简历中怎么写

可以写：

> 在 i7-14700KF 本机使用确定性模型完成 500 次全链路基准，吞吐 178.82 QPS，P99 6.236 ms；真实 DeepSeek 5 次端到端冒烟全部成功，中位延迟 17.247 s。

不要写“系统支持 178 QPS”或“可用性 100%”。前者忽略模型调用，后者缺少持续监控窗口。
